# 产品方案

## 1. 目标

做一个轻量工具，用户输入抖音分享文案、短链或链接后，完成：

1. 提取分享 URL
2. 下载视频文件
3. 抽取音频
4. 转写成文字
5. 保存本地结果

## 2. 非目标

第一版不做这些内容：

- 批量抓取
- 多账号调度
- 代理池
- 验证码处理
- 风控规避
- App 私有接口逆向
- 在线管理后台

原因很简单：这些东西会让项目迅速膨胀，而且风险高、维护成本高。

## 3. 推荐形态

第一阶段只做 CLI。

原因：

- 代码最少
- 最容易调试
- 最适合快速验证
- 后续可以平滑加一层 FastAPI

## 4. 用户流程

```text
用户粘贴分享文案
  -> 提取 URL
  -> 下载视频
  -> 提取 wav 音频
  -> 调用转写模型
  -> 生成 txt
```

## 5. 技术方案

### 下载层

使用 `yt-dlp`，原因：

- 已经处理了很多站点差异
- Python 集成成本低
- 比自己写请求和解析逻辑更稳

### 转写层

使用 `faster-whisper`，原因：

- Python 集成简单
- 模型选择灵活
- 适合单机离线处理

### 音频处理

使用 `ffmpeg`，原因：

- 成熟稳定
- 命令简单
- 几乎是事实标准

## 6. 合规和稳定性策略

不做风控规避，改做下面这些稳态策略：

1. 只处理用户主动提交的单任务
2. 默认低并发，甚至单线程
3. 失败自动重试次数很低
4. 需要登录态时，使用用户自己的 cookies
5. 记录失败原因，便于人工介入

这套策略的核心不是“绕过去”，而是让系统工作在更保守的边界内。

## 7. 代码结构建议

```text
src/douyin_pipeline/
  parser.py        # 从分享文案提取 URL
  downloader.py    # 调用 yt-dlp 下载
  transcriber.py   # ffmpeg + whisper 转写
  cli.py           # 命令行入口
  config.py        # 配置项
```

## 8. MVP 验收标准

满足下面几点，就算第一版成立：

1. 能从分享文案中提取出 URL
2. 能下载单个视频到本地
3. 能把视频转成 txt 文本
4. 失败时给出明确错误信息
5. 项目代码量保持很小，能快速读懂

## 9. 第二阶段扩展

如果第一版跑通，再考虑：

1. FastAPI HTTP 接口
2. 任务状态管理
3. SQLite 历史记录
4. Web 页面
5. 多种 ASR 引擎切换

第一版不要直接做这些。
